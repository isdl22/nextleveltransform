## 1. 작업 컨텍스트

### 1.1 배경
취업 준비생이 자소서를 작성할 때 가장 어려워하는 지점은 두 가지다. (1) 자신의 경험을 어떻게 의미 있게 분해해야 할지 모르고, (2) 같은 경험을 회사·문항마다 어떻게 다르게 풀어내야 할지 모른다. 이 시스템은 **『자소서 바이블 3.0』(이하 자바3.0)**의 검증된 프레임인 **3C4P**를 기반으로, 어드바이저와의 대화로 경험을 구조화된 데이터로 분해하고, 그 데이터를 다양한 자소서 문항에 재활용하는 워크플로우를 자동화한다. 대학을 대상으로 B2B 라이선스 형태로 판매하며, 사용자(학생)·대학 관리자·운영자(개발자) 3계층의 페이지를 제공한다.

### 1.2 목적
- 사용자의 경험을 **3C4P 13개 필드의 구조화된 데이터**로 분해·축적한다.
- 분해된 데이터를 **여러 회사·문항에 재활용**하여 자소서 초안을 생성한다.
- 자소서 유형(필살기 / 지원동기 / 인성·기타)에 따라 적절한 추가 정보를 결합한다.
- 대학 관리자가 학생 진척도를 모니터링하고, 운영자가 프롬프트·품질·퍼널을 관리한다.

### 1.3 범위
**포함**
- 사용자 프로필 입력 (기본 이력 + 지원 직무)
- **채팅 및 음성 입력 방식 모두 지원** — 어드바이저와의 대화는 두 방식 자유롭게 전환 가능
- 경험 리스트업 → 3C4P 분해 (어드바이저 대화)
- 가치관·성격 정보 수집 (인성 문항용)
- 기업 조사 자료(사용자 업로드) 정리
- 자소서 문항 분류 → 유형별 자소서 생성
- 자바3.0 기준 셀프 첨삭
- 사용자 / 대학 관리자 / 운영자 3계층 페이지

**제외**
- 면접 준비, 이력서·포트폴리오 작성
- 기업 정보의 외부 자동 수집(웹 크롤링) — 사용자 업로드 자료에 한정
- **STT(음성→텍스트 변환) 엔진 자체 구축** — 외부 API(Whisper / Google STT / Naver Clova 등) 사용을 전제. 에이전트 시스템은 텍스트화된 입력을 받음

### 1.4 입력 / 출력
| 구분 | 항목 | 형태 |
|------|------|------|
| **입력** | 사용자 프로필 | 자유 텍스트 (대화형) |
| | 경험 진술 | **자유 텍스트 (채팅) 또는 음성→STT 변환 텍스트** |
| | 가치관·성격 정보 | 자유 텍스트 (대화형, 인성 문항 첫 시도 시) |
| | 기업 조사 자료 | 파일 업로드 (PDF, TXT, MD 등) |
| | 자소서 문항 | 회사명 + 문항 텍스트 + (선택) 글자수 |
| **출력** | 사용자 프로필 | `/output/profile.json` |
| | 경험 풀 | `/output/experience-pool/exp-XXX.json` (누적) |
| | 가치관·성격 데이터 | `/output/values-personality.json` |
| | 기업 조사 정리본 | `/output/company-research/<회사명>/summary.json` |
| | 완성된 자소서 | `/output/jasoseo/<회사명>_<문항요약>.md` |

### 1.5 입력 채널: 채팅 vs 음성

두 입력 방식은 모두 동일한 어드바이저와 대화하지만, 처리 특성이 다르므로 시스템 단에서 메타데이터로 구분한다.

| 항목 | 채팅 입력 | 음성 입력 |
|------|----------|----------|
| **처리 경로** | 클라이언트 → 백엔드 → 에이전트 | 클라이언트 → STT API → 백엔드 → 에이전트 |
| **속도** | 즉시 | 1~3초 STT 지연 |
| **정제 필요성** | 낮음 (사용자가 정제해서 입력) | **높음** ("음...", "어...", 말끊김, 문장 미완결) |
| **표현 특성** | 짧고 정형화 경향 | 길고 구어체, 반복·잉여 표현 多 |
| **꼬리질문 트리거** | 표준 | 음성은 답변 길이가 길어도 핵심 부재 가능 → 트리거 강도 조정 |
| **입력 메타데이터** | `source: "chat"` | `source: "voice"` + STT 신뢰도(confidence) |

**처리 원칙**: 어드바이저는 입력 소스(채팅/음성)를 알고 있어야 한다. 음성 입력의 경우 첫 단계에서 **구어체 정제(filler words 제거 + 문장 단위 분할)**를 거친 후 동일한 분해 로직으로 처리한다. 사용자에게는 정제 결과를 한 번 보여주고 확인받는 단계를 둔다(STT 오인식 보정).

### 1.6 제약조건
- **자바3.0 프레임 준수**: 3C4P 13필드 정의, 자소서 구조(소제목+3C+4P+인사이트), 분량 비율을 자료 기준 그대로 따른다.
- **데이터 영속성**: 경험 풀은 한 번 만들면 계속 누적되어 재활용된다. 같은 경험을 두 번 분해하지 않는다.
- **사용자 확인 필수 지점**: ① 음성 입력 정제 결과 ② 자소서 문항 분류 결과 ③ 기업 조사 정리본 ④ 자소서 최종본
- **꼬리질문 한계**: 한 필드당 최대 5회 재시도. 초과 시 사용자에게 스킵 여부 확인.
- **자료 출처 표기**: 자바3.0의 프레임을 그대로 활용하되, 사용자에게 "자바3.0 기반"임을 명시한다.
- **테넌트 격리**: 대학별 데이터는 격리되며, 한 대학 학생이 다른 대학 데이터를 볼 수 없다.

### 1.7 용어 정의
| 용어 | 정의 |
|------|------|
| **3C** | 경험의 **배경(Why)**. Customer(혜택 대상), Company(소속/목표/문제), Competitor(조사 대상/내용) |
| **4P** | 경험에서 **내가 한 것(What+How)**. Product(결과), Place(문제해결 행동), Price(생산성 행동), Promotion(소통·설득 행동) |
| **3C4P 13필드** | 자바3.0 p.105 기준의 13개 데이터 필드. 본 시스템의 **데이터 스키마 단위**. 상세는 §1.8 참조 |
| **필살기** | 직무 KPI에 직결되는 성공 경험. 자소서의 직무 경험 문항에서 핵심 |
| **경험 리스트업** | 3C4P 분해 전, 사용자가 가진 경험을 10개 카테고리로 모으고 KPI 매칭 가능성을 평가하는 단계 |
| **어드바이저** | 사용자와 대화하여 정보를 수집·분해하는 서브에이전트. 경험 어드바이저 / 가치관 어드바이저로 분리 |
| **꼬리질문** | 사용자 답변이 모호할 때 구체화를 유도하는 후속 질문. 트리거 조건 4가지 (수치 부재 / Why 부재 / 액션 모호 / 결과 누락) |
| **테넌트** | 대학 단위. 대학 간 데이터는 완전히 격리된다 |

### 1.8 3C4P 13필드 정의 (자바3.0 p.105 기준)

| # | 카테고리 | 필드명 | 내용 | 자바3.0 페이지 |
|---|---------|--------|------|----|
| 1 | (소제목) | **How + Result** | 4P의 행동(How) + Product(결과)로 구성된 헤드라인 | p.135 |
| 2 | 3C - Customer | 혜택을 받는 대상 | 1차/2차 고객 (B2B의 경우 고객사+최종소비자) | p.106 |
| 3 | 3C - Customer | 대상이 필요로 한 것 | 고객의 니즈, 페인포인트 | p.107 |
| 4 | 3C - Company | 내가 속했던 곳/나의 역할 | 소속 조직, 직책, 담당 영역 | p.110 |
| 5 | 3C - Company | 본인(팀)의 목표 | 당시 달성하려 한 것 | p.110 |
| 6 | 3C - Company | 본인(팀)의 문제 혹은 기회 상황 | 직면한 문제 또는 기회 | p.110 |
| 7 | 3C - Competitor | 조사 대상 | 비교·분석한 대상(경쟁사·유사사례·선행연구) | p.112 |
| 8 | 3C - Competitor | 조사 내용 | 조사한 내용과 인사이트 | p.112 |
| 9 | 4P - Product | 결과 | 정량적 결과 (수치 필수) | p.118 |
| 10 | 4P - Product | 결과의 의미 | 결과가 회사·고객에게 갖는 임팩트 | p.119 |
| 11 | 4P - Place | 문제해결을 위한 행동 | 구체적 문제 해결 / 데이터 분석 / 의사소통·협력 강화 액션 | p.120 |
| 12 | 4P - Price | 생산성을 높인 행동 | 비용 절감 / 업무시간 단축 / 자원·인력 효율화 | p.123 |
| 13 | 4P - Promotion | 소통과 설득을 위한 행동 | 고객 행동 유도 / 정보 시각화 / 참여 독려·지속적 소통 | p.126 |

**필수 필드**: 1, 9 (소제목과 결과는 반드시 채워져야 함). 나머지는 권장.
**수치화 필수**: 9, 12, (가능하면) 11, 13의 결과물.

## 2. 워크플로우 정의

### 2.1 전체 흐름

```
[Phase 1] 프로필 입력
    │
    ▼
[Phase 2] 경험 리스트업 → 3C4P 분해 (반복, 누적)
    │   └ 입력: 채팅 또는 음성(STT 변환)
    ▼
[Phase 3] (선택) 가치관·성격 수집 ── 인성 문항 첫 시도 시
    │
    ▼
[Phase 4] 자소서 작성
    ├─ 4-1. 문항 입력 + LLM 자동 분류 + 사용자 확인
    ├─ 4-2. 분기 처리
    │     ├─ 필살기      → 경험 풀 매칭 → 작성
    │     ├─ 지원동기    → 기업 조사 자료 업로드 → 정리 → 매칭 → 작성
    │     └─ 인성·기타  → 가치관 데이터 + (필요 시) 추가 문답 → 작성
    └─ 4-3. 셀프 첨삭 + 사용자 검토 → 최종 출력
```

### 2.2 단계별 정의

#### Phase 1. 프로필 입력 (1회성, 이후 수정 가능)

| 항목 | 내용 |
|------|------|
| **처리 주체** | 메인 에이전트 (직접 처리) |
| **LLM 판단** | 사용자 자유 텍스트에서 이력 항목 추출 / 누락 항목 식별 |
| **코드 처리** | profile.json 읽기·쓰기 |
| **성공 기준** | 필수 필드(이름, 학력, 지원 직무 카테고리) 모두 채워짐 |
| **검증 방법** | 스키마 검증 (필수 필드 존재 여부) |
| **실패 처리** | 자동 재시도 — 누락 필드만 재질문 (최대 3회) → 초과 시 부분 저장하고 진행 |

#### Phase 2-0. 음성 입력 정제 (음성 입력 시에만)

| 항목 | 내용 |
|------|------|
| **처리 주체** | 메인 에이전트 (스킬 호출: `voice-cleaner`) |
| **LLM 판단** | filler words("음...", "어...") 제거 / 끊긴 문장 복원 / 핵심 의미 보존 |
| **코드 처리** | 입력 메타데이터(`source`, `confidence`) 기록 |
| **성공 기준** | 정제 텍스트가 원문의 핵심 의미를 보존하고, 사용자가 확인 |
| **검증 방법** | 사람 검토 (정제 결과를 사용자에게 보여주고 수정 기회 제공) |
| **실패 처리** | 사용자가 수정 → 수정본 채택. STT 신뢰도가 매우 낮으면 채팅 재입력 권장 |

#### Phase 2-1. 경험 리스트업

| 항목 | 내용 |
|------|------|
| **처리 주체** | 서브에이전트: `experience-advisor` |
| **LLM 판단** | 카테고리별 경험 끌어내기 / 직무 KPI와의 연결 가능성 평가 / 필살기 후보 분류 |
| **코드 처리** | 카테고리 템플릿 로드, experience-list.json 저장 |
| **성공 기준** | 최소 5개 이상 경험 + 각 경험에 KPI 매칭 후보 1개 이상 |
| **검증 방법** | 규칙 기반 (개수, 카테고리 다양성) + LLM 자기 검증 (직무 연결성) |
| **실패 처리** | 자동 재시도 (예시 제공하며 추가 경험 유도, 최대 3회) → 부족하면 진행하되 로그 기록 |

#### Phase 2-2. 경험 선택 + 3C4P 분해

| 항목 | 내용 |
|------|------|
| **처리 주체** | 서브에이전트: `experience-advisor` |
| **LLM 판단** | 13필드 순차 질문 / 답변 모호도 판정 / 꼬리질문 생성 / 예시 매칭 / 답변을 필드값으로 정제 |
| **코드 처리** | 필드 스키마 검증, experience-pool/exp-XXX.json 저장 |
| **성공 기준** | ① 13개 필드 중 필수 필드(#1, #9) 채워짐 ② Product(#9)에 수치 포함 ③ 3C(Why)와 4P(What/How)가 논리적으로 연결됨 |
| **검증 방법** | 스키마 검증(필드 존재) + LLM 자기 검증(수치·논리·Why) + 사람 검토(분해 결과 사용자 확인) |
| **실패 처리** | 필드 누락·모호 → 자동 재시도 (꼬리질문, 필드당 최대 5회) → 초과 시 에스컬레이션 (사용자에게 스킵/수동입력/포기 선택) |

**꼬리질문 트리거 조건 (LLM이 판단)**
- 수치 부재: Product/Place/Price에 정량적 결과 없음
- Why 부재: Customer 니즈 또는 Company 목표 불명확
- 액션 모호: "열심히", "꼼꼼히" 등 추상 표현만 있고 구체 행동 부재
- 결과 누락: 의미(결과의 임팩트) 부재

**음성 입력 시 추가 고려**: 답변이 길어도 핵심 부재 가능 → LLM이 길이 기반이 아닌 **내용 기반**으로 트리거 판정.

#### Phase 3. 가치관·성격 수집 (선택, 인성 문항 처리 시 자동 트리거)

| 항목 | 내용 |
|------|------|
| **처리 주체** | 서브에이전트: `values-advisor` |
| **LLM 판단** | 성장과정 키워드(직무 일관성/특이 경로/가치관) 추출 / 장단점 + 직무 연결 사례 매칭 / MBTI 기반 키워드 추천 |
| **코드 처리** | values-personality.json 저장 |
| **성공 기준** | 성장과정 키워드 ≥ 1개, 장점 1개+사례 2~3개, 단점 1개+극복 사례 1개 |
| **검증 방법** | 규칙 기반(필수 항목 존재) + LLM 자기 검증(컬처핏 적합도) |
| **실패 처리** | 자동 재시도 → 부족 시 자소서 작성 시 즉석 보강 |

#### Phase 4-1. 자소서 문항 입력 + 분류

| 항목 | 내용 |
|------|------|
| **처리 주체** | 메인 에이전트 |
| **LLM 판단** | 문항 텍스트 → 유형 분류(필살기 / 지원동기 / 인성·기타) + 분류 근거 제시 |
| **코드 처리** | 분류 결과 저장 |
| **성공 기준** | 유형 분류 + 사용자 확인 완료 |
| **검증 방법** | 사람 검토 (LLM 분류 결과 + 근거를 사용자에게 보여주고 확인) |
| **실패 처리** | 사용자가 분류 변경 시 그 결정을 따른다 |

#### Phase 4-2a. 필살기 분기

| 항목 | 내용 |
|------|------|
| **처리 주체** | 서브에이전트: `jasoseo-composer` (필살기 모드) |
| **LLM 판단** | 경험 풀에서 문항·직무에 가장 적합한 경험 추천 / 소제목(How+Result) 생성 / 3C(30~40%) + 4P(50~60%) + 인사이트 조립 / 글자수 맞춤 |
| **코드 처리** | experience-pool 읽기, 자소서 파일 저장 |
| **성공 기준** | ① 자바3.0 구조 준수 ② 글자수 ±10% ③ 수치 포함 ④ 소제목 존재 |

| **실패 처리** | 데이터 부족 → 즉석 문답으로 보강 |

#### Phase 4-3. 셀프 첨삭 + 출력

| 항목 | 내용 |
|------|------|
| **처리 주체** | 메인 에이전트 (스킬 호출: `self-review-checker`) |
| **LLM 판단** | 자바3.0 부록 "셀프 자소서 첨삭 체크리스트" 항목별 평가 |
| **코드 처리** | 자소서 파일 저장 |
| **성공 기준** | 체크리스트 통과율 ≥ 80% |
| **검증 방법** | LLM 자기 검증(체크리스트) + 사람 검토(최종본) |
| **실패 처리** | 통과율 미달 → 자동 재작성(최대 2회) → 그래도 미달 시 사용자에게 항목별 피드백 제시 |

### 2.3 LLM 판단 영역 vs 코드 처리 영역

| LLM이 직접 수행 | 스크립트로 처리 |
|---|---|
| 음성 입력 정제 (filler 제거·문장 복원) | STT API 호출, 텍스트 수신 |
| 사용자 답변에서 3C4P 필드 매핑 | JSON 읽기·쓰기, 디렉터리 관리 |
| 답변 모호도 판정 + 꼬리질문 생성 | 스키마 검증(필수 필드 존재) |
| 직무 KPI와 경험의 연결성 평가 | 글자수 카운트 |
| 자소서 문항 유형 분류 | 파일 업로드 파싱 (PDF→텍스트) |
| 경험 추천(문항·직무 적합도) | 경험 풀 인덱싱·검색 |
| 자소서 조립(소제목·3C·4P·인사이트) | 출력 파일명 규칙 적용 |
| 셀프 첨삭 체크리스트 평가 | 체크리스트 점수 집계 |
| 컬처핏 키워드 추천 | 이벤트 로깅(퍼널 트래킹용) |

## 4. 3계층 페이지 구조 (신규 섹션)

본 시스템은 사용자(학생) / 대학 관리자 / 운영자(개발자) 세 계층의 페이지를 제공한다. 각 페이지는 동일한 백엔드(에이전트 시스템 + DB)를 공유하지만, 권한과 기능이 명확히 분리된다.

### 4.1 권한 모델

| 역할 | 접근 가능 데이터 | 주요 기능 |
|------|------------------|----------|
| **학생 (user)** | 본인 데이터만 | 경험 분해, 자소서 작성, 결과 확인 |
| **대학 관리자 (univ_admin)** | 소속 대학 학생들의 통계·자소서 메타데이터 / 권한 부여 시 본문 | 진척도 모니터링, 라이선스 관리 |
| **운영자 (operator)** | 전체 / 대학·학생 식별정보는 마스킹 | 프롬프트 관리, 퍼널 분석, 품질 모니터링 |
| **슈퍼 운영자 (super_admin)** | 전체 + 식별정보 접근 | 권한 부여, 라이선스 발급, 장애 대응 |

**원칙**
- 운영자는 기본적으로 **익명화된 데이터**만 본다. 식별정보 접근은 슈퍼 운영자가 일시 권한 부여한 경우에 한정.
- 대학 관리자는 **자기 대학 학생만** 본다 (테넌트 격리).
- 모든 페이지는 SSO(대학 SAML) 또는 이메일 도메인 기반 자동 라우팅.

### 4.2 사용자 페이지 (학생용)

**핵심: 에이전트와의 대화 + 본인 데이터 관리**

| 영역 | 기능 |
|------|------|
| **온보딩** | 가입, 프로필 입력, 자바3.0 개념 튜토리얼(3C4P가 무엇인지) |
| **어드바이저 채팅** | 채팅·음성 토글, 경험 분해 진행 / 가치관 수집 / 기업 조사 자료 업로드 |
| **마이 데이터** | 경험 풀 목록·검색·수정·삭제 / 가치관 데이터 / 기업 조사 자료 / 프로필 수정 |
| **자소서 작성** | 회사·문항 입력, 분류 확인, 결과 검토, 글자수 조정 |
| **자소서 라이브러리** | 작성한 자소서 목록, **버전 히스토리**, 즐겨찾기 |
| **결과 입력** | 합격/불합격/면접 진행/미응답 — 합격률 데이터 축적용 |
| **익스포트** | PDF / DOCX / 클립보드 복사 |
| **공유 링크** | 멘토·교수에게 링크로 공유, 코멘트 받기 (선택) |
| **알림 설정** | 이메일 / 알림톡 / 인앱 |
| **자동 저장** | 작성 중 데이터 유실 방지 (5초 단위) |

### 4.3 대학 관리자 페이지 (취업지원처용)

**핵심: 학생 진척도 가시화 + 라이선스·운영 관리**

#### 4.3.1 대시보드 (메인)
- **활성 사용자 수**: DAU / WAU / MAU
- **퍼널 진척도**:
  - 가입 → 프로필 완성 → 경험 1건 분해 → 경험 5건 분해 → 자소서 1건 작성 → 자소서 5건 작성
  - 각 단계별 학생 수와 전환율
- **자소서 통계**: 총 작성 수, 유형별(필살기/지원동기/인성) 분포, 회사·산업별 분포
- **라이선스 사용 현황**: 좌석 N개 중 M개 사용, 만료일

#### 4.3.2 학생 관리
- **학생 목록**: 학과·학년·진척 단계 필터, 검색
- **개별 학생 프로필**: 가입일·마지막 활동·경험 분해 개수·자소서 작성 개수
  - 권한 부여 시 자소서 본문 열람 가능
- **그룹 관리**: 학과·학년·취업 캠프 등 그룹화
- **미활동자 리마인드**: 가입 후 N일 활동 없는 학생에게 알림톡·이메일 자동 발송 (템플릿 편집 가능)
- **대량 초대**: 학생 명단 CSV 업로드 → 가입 링크 자동 발송

#### 4.3.3 통계·분석
- **합격률 트래킹**: 학생이 결과 입력한 자소서 기준 회사·직무별 합격률
- **인기 회사·직무 Top N**: 학생들이 많이 지원한 곳
- **경험 카테고리 분포**: 학생들이 어떤 경험을 많이 분해하는지
- **이용 시간대 분포**: 시스템 활성 시간

#### 4.3.4 콘텐츠·운영
- **익명화된 우수 사례**: 합격 자소서를 익명화·승인 후 학생들에게 노출 (학생 동의 필수)
- **공지사항**: 학교 차원의 취업 캠프, 박람회 안내
- **자료실**: 학교가 추가 제공하는 가이드 (학교 양식 자소서 등)

#### 4.3.5 데이터 관리
- **데이터 익스포트**: 학기별·연도별 리포트 CSV/엑셀
- **개인정보 처리**: 졸업생 자동 비활성화, 데이터 삭제 요청 처리
- **개인정보 열람 권한**: 자소서 본문 등 민감 정보 열람 시 사유 기록

### 4.4 운영자 페이지 (개발팀용)

**핵심: 프롬프트 관리 + 퍼널 분석 + 품질 모니터링**

#### 4.4.1 프롬프트 관리 (가장 중요)
- **에이전트별 프롬프트 편집기**: 4개 서브에이전트 + 12개 스킬 각각의 프롬프트 텍스트 관리
- **버전 관리**: 모든 변경 이력 보관, 누가 언제 무엇을 바꿨는지, **롤백 가능**
- **A/B 테스트**: v1/v2를 동시 운영, 사용자 그룹 분할, 결과 비교(품질·재시도율·소요 시간)
- **테스트 샌드박스**:
  - 프롬프트 수정 후 즉석 테스트 — 샘플 사용자 답변 입력 → 에이전트 결과 즉시 확인
  - 실제 사용자 데이터 N건 선택 → 새 프롬프트로 재실행 → 차이 비교
- **프롬프트 라이브러리**: 폐기된 버전, 실험 버전 보관

#### 4.4.2 퍼널 분석
- **단계별 전환율**:
  - 가입 → 프로필 완성률
  - 프로필 → 경험 리스트업 시작률
  - 경험 분해 시작 → 1건 완성률
  - 경험 분해 완성 → 자소서 작성 시작률
  - 자소서 작성 시작 → 완성률
  - 자소서 완성 → 다운로드/공유율
- **이탈 지점 분석**: 어느 단계에서 가장 많이 이탈하는지 + 이탈 직전 마지막 이벤트
- **꼬리질문 통계**: 필드별 평균 꼬리질문 횟수, 5회 초과 케이스
- **세션 길이·재방문**: 코호트 분석

#### 4.4.3 품질 모니터링
- **자소서 품질 자동 검사**: 길이·구조·수치 포함 여부 자동 검사 → 이상 결과 알림
- **셀프 첨삭 점수 분포**: 통과율 80% 미만 케이스 모아보기
- **에러·재시도 로그**: 꼬리질문 5회 초과한 사용자·필드 리스트 → 프롬프트 개선 시그널
- **사용자 피드백**: 별점·코멘트 집계
- **음성 입력 STT 신뢰도 분포**: 낮은 신뢰도 케이스 → STT 엔진 개선 시그널

#### 4.4.4 비용·리소스 모니터링
- **LLM API 비용**: 모델별·에이전트별·대학별·사용자별 일/주/월 집계
- **비용 알림**: 일일 한도 초과 시 슬랙·이메일
- **모델 라우팅 정책**: 분류는 Haiku, 작성은 Sonnet, 첨삭은 Opus 등 정책 관리
- **STT API 비용**: 음성 입력 사용량 + 비용

#### 4.4.5 권한 관리
- **사용자 검색 + 역할 부여/회수**: student / univ_admin / operator / super_admin
- **대학 관리자 임명**: 대학별 univ_admin 계정 발급·회수
- **권한별 기능 매트릭스**: 어떤 역할이 어떤 페이지·기능에 접근 가능한지
- **감사 로그**: 권한 변경 이력, 관리자의 학생 데이터 열람 이력

#### 4.4.6 콘텐츠 관리
- **자바3.0 레퍼런스 업데이트**: 책이 4.0으로 업데이트되면 어디를 손봐야 하는지 가이드
- **카테고리·직무 코드 관리**: 경험 카테고리, 직무 분류 등 마스터 데이터
- **공지사항·이메일 템플릿**: 시스템 차원 알림

#### 4.4.7 운영
- **장애·에러 대시보드**: APM 연동, 실시간 에러 알림
- **DB 상태**: 스토리지 사용량, 백업 상태
- **고객 지원 티켓**: 학생·대학 관리자 문의 처리

### 4.5 페이지별 데이터 흐름

```
[학생 행동] → [에이전트 시스템] → [DB + 이벤트 스트림]
                                          │
                       ┌──────────────────┼──────────────────┐
                       ▼                  ▼                  ▼
              [학생 페이지]        [대학 관리자 페이지]    [운영자 페이지]
              본인 데이터만        본인 대학 통계          전체 + 익명화
```

**이벤트 스트림**: 모든 단계(Phase 1~4의 진입·완료·재시도·에스컬레이션)는 이벤트로 발행되어 퍼널 분석·품질 모니터링에 활용된다. 메인 에이전트가 각 단계 시작/종료 시점에 이벤트를 발행하는 책임을 진다(CLAUDE.md §11).

## 5. 프롬프트 최적화 시스템 (신규 섹션)

본 시스템의 자소서 품질은 어드바이저·작성기의 **프롬프트 품질**에 의해 좌우된다. 시간이 흐를수록 데이터가 누적되고 더 좋은 프롬프트로 진화해야 하며, 이를 자동화하는 것이 본 섹션의 목표다.

### 5.1 목표와 한계

**목표**: 사용자 사용·결과 데이터를 기반으로 프롬프트가 지속 최적화되어, 자소서 품질이 시간에 따라 우상향한다.

**한계 인식 (합격률을 직접 학습 신호로 쓰기 어려운 이유)**

| 한계 | 설명 |
|------|------|
| 신호 희소성 | 학생 1명이 자소서 5개 쓰고 그중 합격 1~2개. 의미 있는 학습 데이터 누적까지 시간 大 |
| 피드백 지연 | 작성 → 서류 합격 → 면접 → 최종 합격까지 1~3개월 |
| 혼란 변수 | 합격은 자소서만의 결과 아님 (학교·학점·면접·운). 인과 입증 어려움 |
| 재현 불가 | 같은 자소서를 100번 제출해 합격률 측정 불가 (일회성 데이터) |

**대응 원칙**: 합격률을 직접 reward로 쓰는 fine-tuning/RLHF는 데이터가 충분히 누적된 후(수만 건)에야 의미가 있다. 그 전까지는 (1) LLM Judge 점수, (2) 합격 사례 RAG, (3) A/B 테스트, (4) 운영자 큐레이션을 조합해 같은 효과를 낸다.

### 5.2 4단계 로드맵 (데이터 누적에 따른 단계적 도입)

| Phase | 도입 시점 | 핵심 메커니즘 | 자동화 수준 |
|-------|----------|--------------|------------|
| **A. A/B + LLM Judge** | MVP 출시 시점 | 프롬프트 v1/v2 분할 노출 + 자바3.0 셀프 첨삭 자동 채점 | 운영자 의사결정 |
| **B. 합격 사례 RAG** | 합격 사례 100~500건 누적 후 (3~6개월) | 합격 자소서를 익명화·구조화 → 새 자소서 작성 시 retrieval | 자동 컨텍스트 주입 |
| **C. DSPy 자동 최적화** | 라벨 데이터 충분 후 (6~12개월) | 프롬프트를 학습 가능한 파라미터로 취급, LLM Judge + 합격률 결합 reward로 자동 튜닝 | 진짜 자동 최적화 |
| **D. Fine-tuning (선택)** | 데이터 1만 건 이상 (장기) | 자체 모델 fine-tuning | 비용 대비 효과 신중 평가 |

**중요**: Phase B+C 조합으로 대부분 충분하다. Phase D까지 가지 않아도 되는 경우가 많다.

#### Phase A 상세

| 항목 | 내용 |
|------|------|
| **A/B 테스트 구성** | 사용자 그룹을 N:N으로 분할, 같은 어드바이저·스킬에 대해 v1/v2 동시 운영 |
| **분할 단위** | 사용자 단위(같은 사용자는 항상 같은 버전) — 일관된 경험 보장 |
| **평가 신호** | LLM Judge 점수 / 꼬리질문 횟수 / 사용자 만족도(별점) / 자소서 완성률 / 작성 소요 시간 |
| **합격률** | 누적되면 보조 신호로 사용 (직접 학습 reward는 아님) |
| **의사결정** | 운영자가 데이터 보고 판단, **수동 채택·롤백** |
| **외부 도구 옵션** | Langfuse(오픈소스), PromptLayer, Helicone — 빠른 도입을 위해 권장 |

#### Phase B 상세 (가장 실용적인 "합격률 기반 학습"의 형태)

| 항목 | 내용 |
|------|------|
| **수집 대상** | 학생이 결과 입력 시 합격(서류·최종)으로 표기한 자소서 |
| **수집 동의** | 가입 시 약관 + 자소서별 별도 옵트인 (학생이 매 자소서마다 익명화 공개 동의) |
| **익명화** | 회사명·고유 정보·식별 가능 표현 제거. 직무·산업·문항 유형은 보존 |
| **구조화** | 자소서를 다시 13필드 구조 + 메타데이터(직무·산업·글자수·합격 유형)로 분해 |
| **활용 방식** | 새 자소서 작성 시 같은 직무·산업·문항 유형의 합격 사례를 retrieval → composer 프롬프트에 컨텍스트 주입 |
| **자바3.0 연계** | 자료의 "직무 계열별 케이스 뱅크" 개념과 동일. 책의 정적 사례 + 우리 시스템의 동적 누적 사례 |
| **품질 관리** | 운영자가 합격 사례 큐레이션 (이상 케이스·중복·품질 미달 제외) |

#### Phase C 상세

| 항목 | 내용 |
|------|------|
| **프레임워크** | DSPy (Stanford 오픈소스) — 프롬프트를 학습 가능한 파라미터로 취급 |
| **학습 데이터** | LLM Judge 고득점 자소서 + 합격 자소서 (Phase B에서 누적) |
| **최적화 목표** | 단일 reward가 아닌 가중 합산: LLM Judge 점수(70%) + 합격률(20%) + 사용자 만족도(10%) — 가중치는 운영자 조정 |
| **운영자 역할** | "목표 메트릭"만 정의, 프롬프트는 시스템이 자동 튜닝 |
| **검증 절차** | 자동 최적화 결과는 항상 A/B 테스트로 기존 버전과 비교 후 채택 |

### 5.3 합격 사례 DB 스키마

```
case_bank/
├── case_<id>.json          # 익명화된 합격 자소서
└── index.json              # 직무·산업·유형별 인덱스
```

**case_<id>.json 핵심 필드**

| 필드 | 설명 |
|------|------|
| `id` | 사례 ID |
| `job_category` | 직무 대분류 (영업/마케팅/개발/생산/연구/기획 등) |
| `job_detail` | 직무 세부 |
| `industry` | 산업 (반도체/금융/유통 등) |
| `company_size` | 대기업/중견/공기업/스타트업 |
| `question_type` | 필살기/지원동기/성장과정/장단점/기타 |
| `question_text` | 문항(원문 또는 의역) |
| `char_limit` | 글자수 제한 |
| `body` | 자소서 본문 (익명화 완료) |
| `decomposed_3c4p` | 13필드로 재분해한 구조 (Phase B의 retrieval 대상) |
| `result` | 서류 합격 / 최종 합격 |
| `quality_score` | LLM Judge 점수 |
| `consent_at` | 학생 동의 시점 |
| `submitted_at` | 자소서 제출 시점 |
| `created_at` | DB 등록 시점 |

### 5.4 LLM-as-Judge 평가 항목 (Phase A부터 적용)

자바3.0 부록 "셀프 자소서 첨삭 체크리스트"를 LLM Judge용 항목으로 구조화한다.

| 카테고리 | 평가 항목 | 가중치 |
|---------|----------|--------|
| **구조** | 소제목 존재 / How+Result 형태 | 10% |
| | 자바3.0 분량 비율 준수 (필살기 3C 30~40% / 4P 50~60%) | 10% |
| **수치화** | Product에 정량 결과 포함 | 15% |
| | 4P 행동에 수치 동반 | 10% |
| **논리** | 3C(Why) → 4P(What/How) 흐름 명확 | 15% |
| | 비약 없음 (액션과 결과의 연결) | 10% |
| **표현** | 추상 수식어("열심히", "꼼꼼히") 최소화 | 5% |
| | 두괄식 |  5% |
| **적합성** | 직무 KPI와의 연결성 | 10% |
| | 글자수 ±10% 준수 | 10% |

**Judge 모델**: Claude Sonnet 또는 Opus를 채점자로 사용. 모델 자체의 평가 일관성을 위해 같은 자소서를 3회 채점 후 평균(self-consistency)을 사용하는 옵션 검토.

### 5.5 운영자 페이지에서의 노출 (§4.4 보강)

§4.4.1 "프롬프트 관리"는 다음 기능을 포함한다.

| 영역 | 기능 |
|------|------|
| **프롬프트 편집기** | 4개 서브에이전트 + 12개 스킬의 프롬프트 텍스트 편집 |
| **버전 관리** | 변경 이력·작성자·롤백 |
| **A/B 실험 관리** | v1/v2 분할 비율 설정, 실시간 결과 비교 |
| **테스트 샌드박스** | 샘플 입력으로 즉석 결과 확인 / 실제 사용자 데이터 N건으로 재실행·차이 비교 |
| **LLM Judge 대시보드** | 카테고리별 평균 점수 추이, 항목별 약점 식별 |
| **합격 사례 큐레이션** | 학생이 합격 표기한 자소서 검토·승인·익명화 보정·DB 등록 |
| **자동 최적화 결과 검토 (Phase C)** | DSPy가 제안한 새 프롬프트 비교·승인 |
| **데이터 수집 동의 현황** | 학생별 옵트인 상태, 동의 철회 처리 |

### 5.6 데이터 수집·동의 정책

합격 사례 DB는 학생 데이터를 활용하므로 동의·익명화 정책이 핵심이다.

| 단계 | 정책 |
|------|------|
| **가입 시 약관** | 자소서 데이터의 익명화 활용 가능성 명시 (단, 자동 동의는 X) |
| **자소서별 옵트인** | 합격 결과 입력 시 "이 자소서를 익명화해 다른 학생을 돕는 데 활용해도 좋다"는 별도 동의 |
| **익명화 검증** | 자동 익명화(회사명·기관명·이름 등 NER 기반 마스킹) 후 운영자 수동 검토 |
| **동의 철회** | 학생이 언제든 철회 가능 → 해당 사례 즉시 DB에서 제거 |
| **대학 관리자 권한** | 본인 대학 학생들의 합격 사례 통계는 볼 수 있으나, 다른 학생의 자소서 본문은 별도 권한 시에만 |
| **법적 검토** | PIPA(개인정보보호법) 준수, 가명정보 처리 가이드라인 적용 |

### 5.7 단계 전환 기준

| 전환 | 기준 |
|------|------|
| **A → B** | 누적 사용자 500명 이상 + 합격 사례 100건 이상 + 동의율 30% 이상 |
| **B → C** | 합격 사례 1,000건 이상 + LLM Judge 점수 데이터 1만 건 이상 |
| **C → D 검토** | 데이터 1만 건 이상 + Phase B+C로도 한계가 명확할 때 |

각 전환 시점은 데이터 누적 속도에 따라 가변. 예상보다 빠르면 앞당기고, 느리면 Phase A에서 더 머문다.

## 6. 핵심 설계 결정 요약

| 결정 사항 | 선택 | 이유 |
|-----------|------|------|
| 입력 채널 | 채팅 + 음성 모두 | 작업 개요에 명시. STT는 외부 API |
| 데이터 영속성 | 누적·재활용 | 같은 경험 여러 회사·문항에 활용 |
| 기업 조사 수집 | 사용자 업로드 + LLM 정리 | 자동 크롤링 정확도 리스크 |
| 자소서 문항 분류 | LLM 자동 + 사용자 확인 | 자동화 + 분기 결정의 중요성 |
| 에이전트 구조 | 메인 + 서브에이전트 4개 | 도메인 분리 + 컨텍스트 최적화 |
| 페이지 구조 | 사용자/대학 관리자/운영자 3계층 | B2B SaaS 표준 |
| 운영자 권한 | 기본 익명화 + 권한별 식별정보 접근 | 개인정보 보호 |
| 프롬프트 관리 | 운영자 페이지에서 버전·A/B·롤백 | 지속적 품질 개선 |
| 프롬프트 최적화 | 4단계 로드맵(A/B → RAG → DSPy → FT) | 합격률 직접 학습은 신호 희소·지연·혼란변수 한계. 단계별 도입이 현실적 |
| 합격 사례 활용 | RAG 방식 (Phase B) | 학습보다 retrieval이 데이터 적게 필요·즉시 반영 |
| 학생 데이터 동의 | 옵트인 + 자소서별 별도 동의 + 운영자 익명화 검증 | PIPA 준수, 신뢰 확보 |

## 7. 리스크 및 대응

| 리스크 | 대응 |
|--------|------|
| 음성 입력 STT 오인식으로 분해 결과 오류 | 정제 결과 사용자 확인 단계 의무화 + 신뢰도 낮으면 채팅 권장 |
| 사용자가 너무 짧게 답변 → 꼬리질문 무한 반복 | 필드당 5회 한계 + 스킵 옵션 |
| 경험 풀이 적은데 자소서 작성 시도 | Phase 2로 우선 안내, 강제는 X |
| 동일 회사에 여러 자료 업로드 시 중복·충돌 | summary.json은 항상 최신 자료 기준 재생성, raw/ 보관 |
| 자소서 문항 유형 모호(복합) | LLM이 복합 유형으로 분류 + 분기 결합 처리 |
| 자바3.0 가이드 vs 사용자 선호 충돌 | 자바3.0 기본 + 명시적 변경 요청 시에만 조정 |
| 경험 풀 누적 시 검색 비용 증가 | data-store에서 직무·태그 인덱싱 |
| 대학 관리자가 학생 자소서 무단 열람 | 본문 열람 시 사유 기록·로그·학생 동의 정책 |
| 프롬프트 변경으로 품질 저하 | 운영자 페이지 A/B 테스트 의무화, 롤백 가능 |
| 합격 사례 데이터 편향 | 특정 산업·직무에 합격 사례 편중 시 retrieval 결과 편향 → 운영자 큐레이션으로 균형 유지 |
| 익명화 누락으로 개인정보 노출 | 자동 익명화 + 운영자 수동 검토 이중 장치 |
| Phase C 자동 최적화 결과 품질 미달 | A/B 테스트 통과 못한 결과는 자동 폐기, 운영자 승인 의무 |
| 운영자가 식별정보로 사용자 추적 | 기본 익명화, 식별정보 접근은 슈퍼 운영자 일시 권한 |

## 8. 미해결·후속 과제 (구현 단계에서 결정)

### 8.1 에이전트 시스템
- 각 SKILL.md / AGENT.md / CLAUDE.md의 구체적 프롬프트 문구
- 13필드의 JSON 스키마 상세 (필드명·타입·필수 여부·검증 규칙)
- 자바3.0 셀프 첨삭 체크리스트의 항목별 가중치
- PDF 처리 라이브러리 선택 (pypdf vs pdfplumber)
- 경험 풀 검색 알고리즘 (단순 태그 vs 임베딩)
- 자소서 출력 비율 자동 검증 방식
- 음성 입력 정제의 보존 기준 (어디까지 정제하고 어디까지 원문 유지)

### 8.2 페이지·인프라 (별도 설계 문서로 분리 권장)
- 인증·SSO 구현 방식 (자체 vs WorkOS·Auth0)
- 결제 시스템 (B2B 라이선스 모델, 포트원 연동)
- DB 스키마 (테넌트 격리, 이벤트 스트림 테이블)
- 프롬프트 관리 시스템 아키텍처 (버전·A/B·테스트 샌드박스)
- 퍼널 이벤트 스키마
- 익명화 정책 (어떤 필드를 어떻게 마스킹할지)
- ISMS-P 인증 준비
- 호스팅·리전 (한국 대학 대상 → 국내 리전)

### 8.3 프롬프트 최적화 시스템
- Phase A 외부 도구 선정 (Langfuse vs PromptLayer vs Helicone vs 자체 구축)
- LLM Judge 프롬프트 자체의 검증 방식 (사람 평가와의 일치도)
- LLM Judge 채점 일관성 확보 (self-consistency 적용 여부)
- 합격 사례 익명화 자동화 수준 (NER 모델 선택, 마스킹 룰)
- DSPy 도입 시점의 데이터 라벨링 정책
- 단계 전환 기준의 정량 지표 검증 (실제 데이터 추이 보고 조정)
- 자동 최적화의 안전장치 (자동 배포 vs 운영자 승인 의무)
